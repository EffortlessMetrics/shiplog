# shiplog 0.8.0 - Release Readiness

**Tag:** `v0.8.0`
**Date:** 2026-05-15
**Theme:** Evidence repair loop

This readiness ledger is for cutting `v0.8.0` after the evidence repair loop
lane. The short handoff is
[`RELEASE_HANDOFF_0.8.0.md`](../../RELEASE_HANDOFF_0.8.0.md).

## Release Thesis

shiplog 0.8.0 makes a rough first packet improvable. The 0.6/0.7 front door
already produced honest reports; 0.8 adds receipt-derived repair actions, a
copyable local journal repair path, repair-state diffing across reruns, and a
product proof that the packet actually improves.

This is not a second source classifier. Repair commands consume
`intake.report.json` receipts and do not inspect `intake.report.md`, query live
providers, mutate provider records, or require secret values in repair actions.

## What Ships

- `intake.report.json` includes compatible v1 `repair_items` derived from
  existing report receipts.
- `shiplog repair plan --latest` prints the latest report's repair queue.
- `shiplog journal add --from-repair <repair_id>` appends local manual evidence
  for `journal_add` repair items.
- `shiplog repair diff --latest` compares recent repair states by stable
  `repair_key`.
- `docs/guides/evidence-repair-loop.md` teaches the loop without restating the
  spec.
- The front-door product proof covers cold intake, repair plan, journal repair,
  rerun, cleared diff, and improved packet evidence.

## User-Facing Contract

- `shiplog intake --last-6-months --explain` still creates a defensible first
  packet and durable report from an empty directory.
- `shiplog repair plan --latest` reads the latest compatible intake report and
  prints stable, copyable repair guidance.
- `shiplog journal add --from-repair <repair_id>` acts only on report items whose
  action kind is `journal_add`; it writes local `manual_events.yaml` evidence.
- `shiplog repair diff --latest` reports `Cleared`, `New`, `Still open`, and
  `Changed` repair keys between the latest two compatible reports.
- Legacy compatible v1 reports without `repair_items` remain readable; repair
  commands tell the user to rerun intake for repair data.

## Tag And GitHub Release

```bash
git switch main
git pull --ff-only
git tag v0.8.0
git push origin v0.8.0
```

`release.yml` builds the multi-platform binaries and creates a draft GitHub
release. Wait for asset upload and checksum generation before publishing the
draft.

## Publish To crates.io

Publishing requires explicit approval and `CARGO_REGISTRY_TOKEN`.

```bash
CARGO_REGISTRY_TOKEN=<token> cargo publish -p shiplog --locked
```

Do not publish `shiplog-testkit` or `xtask`; they remain `publish = false`.

## Publish GitHub Release

After `release.yml` has uploaded release assets and crates.io publish has
completed, publish the draft GitHub release:

```bash
gh release view v0.8.0
gh release edit v0.8.0 --draft=false --latest
```

## Known Non-Blockers

- The 0.8 release does not add provider mutation. Provider setup remains an
  operator action followed by a rerun.
- Historical 0.6 implementation crates remain available as 0.6 artifacts; 0.8
  continues the 0.7 supported forward public surface of `shiplog` only.

## Rollback Path

- Before tagging: revert the release-prep PR or forward-fix release notes and
  version metadata.
- After tagging but before crates.io publish: delete `v0.8.0` locally and on
  origin, then rerun release prep.
- After crates.io publish: do not delete or rewrite the release. Ship a
  forward-fix patch/minor release and mark the GitHub release notes with the
  known issue.
