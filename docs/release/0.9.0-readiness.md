## Release Thesis

shiplog 0.9.0 makes a repaired packet review-ready. The 0.8 Evidence Repair
Loop made rough packets improvable; 0.9 adds conservative quality surfaces on
top of that loop: packet readiness, evidence strength, receipt-backed claim
candidates, missing-context prompts, share posture explanation, quality diffs,
and a worked guide.

This release does not write performance-review prose, score employees, require
LLMs, mutate provider records, or turn share profiles into dashboards. It gives
the user a defensible scaffold and shows what still needs human context.

## What Ships

- `intake.report.json` includes optional compatible v1 `packet_quality` with
  packet readiness, evidence strength, claim candidates, and share posture
  compatibility fields.
- `packet.md` opens with `Packet Readiness` and renders `Claim Candidates` when
  receipts support them.
- `shiplog share explain manager|public` explains included, removed, blocked,
  and needs-review share posture without requiring a redaction key or writing
  profile artifacts.
- `shiplog runs diff --latest` compares packet quality movement across recent
  runs.
- `docs/guides/review-ready-packet.md` teaches the full collect, repair, rerun,
  compare, interpret, share flow.
- The front-door product proof now covers repaired packet readiness, claim
  candidates, quality diff, and share posture explanation.

## User-Facing Contract

- `shiplog intake --last-6-months --explain` still creates a first packet and
  durable report from an empty directory.
- `shiplog repair plan --latest` and `shiplog journal add --from-repair
  <repair_id>` remain the safe local repair path.
- `shiplog repair diff --latest` shows repair-state movement by stable
  `repair_key`.
- `shiplog runs diff --latest` shows packet-quality movement across reruns.
- `packet.md` can explain readiness and claim candidates without inventing
  impact or final review prose.
- `shiplog share explain manager|public --latest` is read-only and may be used
  before a redaction key exists.
- Manager/public rendering still fails closed without `--redact-key` or
  `SHIPLOG_REDACT_KEY`.

## Publish To crates.io

Publishing requires explicit approval and a Cargo registry credential.

```bash
CARGO_REGISTRY_TOKEN=<token> cargo publish -p shiplog --locked
```

Do not publish `shiplog-testkit` or `xtask`; they remain `publish = false`.

## Publish GitHub Release

After `release.yml` uploads release assets and crates.io publish completes,
confirm the GitHub release is non-draft, non-prerelease, and marked latest:

```bash
gh release view v0.9.0
gh release edit v0.9.0 --draft=false --latest
```

## Known Non-Blockers

- 0.9 does not add provider mutation, OAuth, dashboards, GUI/TUI, plugin APIs,
  or LLM-written prose.
- Public share verification remains a guardrail, not a guarantee of perfect
  privacy.
- Historical 0.6 implementation crates remain available as historical artifacts;
  0.9 continues the 0.7 supported forward public surface of `shiplog` only.

## Rollback Path

- Before tagging: revert the release-prep PR or forward-fix release notes and
  version metadata.
- After tagging but before crates.io publish: delete `v0.9.0` locally and on
  origin, then rerun release prep.
- After crates.io publish: do not delete or rewrite the release. Ship a
  forward-fix patch/minor release and mark the GitHub release notes with the
  known issue.
