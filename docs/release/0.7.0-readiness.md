# shiplog 0.7.0 - Release Readiness

**Tag:** `v0.7.0`
**Date:** 2026-05-14
**Theme:** Crate-surface contraction

This readiness ledger is for cutting `v0.7.0` after the crate-surface
contraction lane. The short handoff is
[`RELEASE_HANDOFF_0.7.0.md`](../../RELEASE_HANDOFF_0.7.0.md).

## Release Thesis

shiplog 0.7.0 keeps the trustworthy first-run review-pack path from 0.6.0 and
makes the package architecture honest: `shiplog` is the supported public crate,
while the former implementation microcrates are historical 0.6 surfaces and
internal SRP modules going forward.

## What Ships

- The default 0.7 publish allowlist contains only `shiplog`.
- `shiplog-schema` is internal for 0.7; JSON schemas under
  `contracts/schemas/` remain the public machine contract.
- Renderers, source adapters, cache/freshness, bundle, redaction, coverage,
  engine, merge/team/workstream/LLM support, IDs, schema, and ports are modules
  inside `shiplog`.
- Release tooling fails closed if unsupported workspace packages or historical
  0.6 implementation crates re-enter the forward publish graph.
- The 0.6 first-run user path remains the product guardrail:

```bash
shiplog intake --last-6-months --explain
shiplog open intake-report --latest
shiplog open packet --latest
```

## Tag And GitHub Release

```bash
git switch main
git pull --ff-only
git tag v0.7.0
git push origin v0.7.0
```

`release.yml` builds the multi-platform binaries and creates a draft GitHub
release. Wait for the workflow to upload assets before publishing the draft.

## Publish To crates.io

Publishing requires explicit approval and `CARGO_REGISTRY_TOKEN`.

```bash
CARGO_REGISTRY_TOKEN=<token> cargo publish -p shiplog --locked
```

Do not run the 0.5 or 0.6 publish scripts for this release. The 0.7 publish
surface is the allowlisted `shiplog` package only.

## Publish GitHub Release

After `release.yml` has uploaded the release assets and crates.io publish has
completed, publish the draft GitHub release:

```bash
gh release view v0.7.0
gh release edit v0.7.0 --draft=false --latest
```

`scripts/verify-release.sh` expects the GitHub release to be non-draft and
non-prerelease.

## Post-Tag Smoke

```bash
scripts/release-install-smoke.sh v0.7.0
scripts/release-install-smoke.ps1 -Version v0.7.0
scripts/verify-release.sh v0.7.0
```

## Known Non-Blockers

- The Evidence Repair Loop remains paused until after 0.7 release closure.
- 0.6 implementation crates remain on crates.io for historical dependency
  resolution and are not yanked absent a security or severe correctness issue.
- `source` remains as a report v1 compatibility alias next to `source_key`.

## Rollback Path

- Before tagging: revert the release-prep PR and keep the crate-surface
  contraction work on `main`.
- After tagging but before crates.io publish: delete `v0.7.0` locally and on
  GitHub, then rerun release prep.
- After crates.io publish: prefer a forward fix (`0.7.1`) unless there is a
  security or severe correctness issue that justifies yanking.
