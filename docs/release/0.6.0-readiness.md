## Release Thesis

review pack with clear output locations, latest-artifact opening, source
state, freshness posture, repair guidance, and proof-backed docs/tests.

## User-Facing Contract

- `shiplog intake --last-6-months --explain` prints the review-pack output
  directory and concise next-step commands.
- `shiplog open intake-report --latest`, `shiplog open packet --latest`, and
  `shiplog open out --latest` resolve the newest run deterministically or tell
  the user how to create one.
- `intake.report.md` keeps reviewer-facing source labels and now includes
  skipped-source freshness rows.
- `intake.report.json` exposes canonical `source_key` plus `source_label`
  for source-facing arrays. The legacy `source` field remains for v1 readers.
- Freshness can be `fresh`, `cached`, `skipped`, `unavailable`, or `stale`.
  `stale` is emitted only after `ApiCache::lookup` returns
  `CacheLookup::Stale(_)`.
- Sharing remains fail-closed under the existing redaction profile contract;
  this release does not reopen protected-fields or `disallowed_fields` work.

## Pre-Tag Commands

Run these on the merged `main` commit before creating `v0.6.0`:

```bash
cargo test -p shiplog --test intake_cold_start
cargo test -p shiplog --test front_door_first_pack_smoke
cargo test -p shiplog --test cli_integration -- intake
cargo test -p shiplog --test cli_integration -- report
cargo test -p shiplog-cache
cargo test -p shiplog-ingest-github
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
cargo xtask package-version
cargo xtask check-policy-schemas
cargo xtask check-file-policy --mode blocking-allowlist
git diff --check
```

On Windows, `cargo xtask package-version` currently delegates to the
`scripts/package-version-audit.sh` bash/python checker. If bash is unavailable,
use `cargo metadata --format-version 1 --no-deps` and verify that every
workspace package is `0.6.0` and every normal workspace dependency requirement
is `^0.6.0`.

## Tag And Release

Do not tag until the release-prep PR is merged and `main` CI is green.

```bash
git switch main
git pull --ff-only
git tag v0.6.0
git push origin v0.6.0
```

The tag triggers `.github/workflows/release.yml` to build release artifacts and
publish the GitHub release.

## Publish Order

Publishing still requires explicit operator approval and `CARGO_REGISTRY_TOKEN`.
The dependency order is encoded in `scripts/publish-v0.6.0.sh`.

```bash
CARGO_REGISTRY_TOKEN=<token> bash scripts/publish-v0.6.0.sh
```

## Known Non-Blockers

- Protected-fields and `disallowed_fields` remain out of scope for this lane.
- The `source` compatibility alias remains in intake-report v1 JSON.
- The active goal should be archived after the release ships or is superseded.

## Rollback Path

- Before tagging: revert the release-prep PR only.
- After tagging but before publishing crates: delete `v0.6.0` locally and on
  origin, then revert or forward-fix.
- After a partial crates.io publish: yank only the affected `0.6.0` crates and
  cut a `0.6.1` or `0.7.0` fix as appropriate.
