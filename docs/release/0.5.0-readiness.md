## Publish dry-run status

There are **two** publish-shaped gates with different semantics:

### `scripts/package-proof.sh` (pre-tag, full workspace)

Runs `cargo package --list -p <crate>` for every publishable crate.
Validates that the packaged file set is clean (no `target/`, no
uncommitted noise, no missing files referenced from `Cargo.toml`). Does
**not** resolve transitive workspace dependencies from crates.io.

```
PASS — package-proof.sh exit 0 on clean tree (this PR's commit).
```

### `scripts/publish-dry-run.sh` (interleaved with publish)

Runs `cargo publish --dry-run -p <crate>` per crate in dependency order.
This gate intentionally hits crates.io to resolve transitive workspace
versions. **Foundation crates pass standalone; non-foundation crates
require the prior crate to be published first** (otherwise the
transitive `shiplog-ids = "^0.5.0"` requirement cannot be satisfied).

```
PASS — shiplog-ids (foundation, no internal deps).
PENDING (post-publish-of-prior-crate) — every other publishable crate.
```

The script accepts `--from <package>` so it can be resumed mid-publish:
publish `shiplog-ids` first (after `cargo publish --dry-run` succeeds
above), then `scripts/publish-dry-run.sh --from shiplog-schema` becomes
green, etc. This is the canonical use pattern.

### Pre-tag commands (run on the merged `main` commit before tagging)

```bash
scripts/package-proof.sh                                # full workspace
cargo publish -p shiplog-ids --dry-run                  # foundation
```

The full `scripts/publish-dry-run.sh` becomes meaningful only after the
foundation crate is published; it is the publish driver, not a pre-tag
gate.

## Tag / publish order

1. Merge PR #157 (this PR) into `main`.
2. Verify `main` CI is green: `Check (ubuntu)`, `Check (windows)`,
   `MSRV (1.95)`, `cargo-deny`, `forecast` (PR Plan), three smokes
   (`bdd-smoke`, `property-smoke`, `fuzz-smoke`), `advisory` (ripr),
   `CodeRabbit`, `GitGuardian`, `droid-review`.
3. Verify `ci-actuals.yml` recorded actuals for the `main` push (joins to
   `policy/ci-lanes.toml`).
4. `git tag v0.5.0` on the merge commit; `git push origin v0.5.0`.
5. Wait for `release.yml` to complete (multi-platform build + checksum +
   release validation).
6. Run `scripts/release-install-smoke.sh v0.5.0` and
   `scripts/release-install-smoke.ps1 -Tag v0.5.0`.
7. Publish to crates.io in dependency order (foundation → adapters →
   engine → CLI):
   ```
   shiplog-ids
   shiplog-schema
   shiplog-ports
   shiplog-coverage
   shiplog-cache
   shiplog-redact
   shiplog-bundle
   shiplog-workstreams
   shiplog-render-md
   shiplog-render-json
   shiplog-cluster-llm
   shiplog-ingest-github
   shiplog-ingest-git
   shiplog-ingest-json
   shiplog-ingest-manual
   shiplog-ingest-gitlab
   shiplog-ingest-jira
   shiplog-ingest-linear
   shiplog-team
   shiplog-merge
   shiplog-engine
   shiplog
   ```
   Each `cargo publish -p <crate>` should be preceded by
   `cargo publish -p <crate> --dry-run` if not already proven by
   `scripts/publish-dry-run.sh`.
8. After all crates are uploaded, announce the release (GitHub release
   notes auto-generated by `release.yml`, plus crates.io listings
   propagating).

## Rollback path

- **Pre-tag**: revert PR #157 on `main`; the v0.4.0 ↔ v0.5.0 surface
  contracts (schemas, ledgers, lanes, readiness doc) are removable in a
  single revert because PR #157 only adds versions/CHANGELOG/docs and
  re-runs proof scripts.
- **Pre-publish, post-tag**: delete the `v0.5.0` tag (`git push --delete
  origin v0.5.0`), undo the GitHub release, leave `Cargo.toml` versions
  as-is (next attempt re-tags the same SHA).
- **Post-publish (partial)**: any subset of crates that did not yet upload
  to crates.io can be retried; uploaded crates cannot be removed but
  `cargo yank --version 0.5.0 -p <crate>` removes them from the
  default-version pool. A 0.5.1 patch can then re-publish if the issue is
  contained.
- **Post-publish (full)**: yank all v0.5.0 crates; immediately cut a
  0.5.1 with the fix (or 0.6.0 if the fix is breaking).
