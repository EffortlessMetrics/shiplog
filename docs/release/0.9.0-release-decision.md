# shiplog 0.9.0 - Release Decision

**Date:** 2026-05-17
**Updated:** 2026-05-20
**Decision:** resume release after owner approval and final merged-main preflight
**Latest shipped release:** `v0.8.0` until `v0.9.0` publish completes

`main`

## Decision

Resume `v0.9.0` release execution.

Owner approval to proceed with `v0.9.0` release execution was recorded on
2026-05-20. The former hold receipt was moved from
`0.9.0-release-hold.md` to
[`0.9.0-release-hold-lifted.md`](0.9.0-release-hold-lifted.md), which removes
the path-based block used by `scripts/check-release-hold.sh`.

This decision does not claim release execution has already happened. The tag,
release workflow, crates.io publish, GitHub release publication, and install
smoke still happen after this release-resume PR merges and current final
preflight passes from `main`.

The product loop is feature-complete enough to release: setup preflight, status
cockpit, intake, repair, rerun, repair diff, runs diff, share explanation, and
budgeted GitHub activity harvest now form one receipt-backed operating loop.

- crates.io latest remains `shiplog = "0.8.0"` until the approved publish step
  completes.
- `v0.9.0` is approved for final preflight and release execution after this
  release-resume PR merges.
- `scripts/check-release-hold.sh v0.9.0` should pass after the hold receipt is
  moved out of `docs/release/0.9.0-release-hold.md`.
- Manual `release.yml` dispatch still requires a semver tag and
  `owner_approved_release_execution`.

## Final Preconditions Before Tagging

- the release-resume PR is merged into `main`;
- current `main` CI is green;
- package-version, package-boundary, publish dry-run, product proof, docs/share
  and status/runs CLI tests, clippy, policy gates, no-panic, generated checks,
  and install smoke planning are rerun from current `main`;
- no local or remote `v0.9.0` tag exists before the approved release action;
- no `v0.9.0` GitHub release exists before the release workflow creates the
  draft release.

## What To Do Next

Do not add features just to justify 0.9.

Allowed next work:

- merge the release-resume PR;
- run final preflight from clean merged `main`;
- tag `v0.9.0` only after final preflight is green;
- let `release.yml` build the draft release and assets;
- publish `shiplog 0.9.0` to crates.io;
- publish the GitHub release and run install smoke.

Do not extend Guided Setup / Doctor, Review-loop Status, or GitHub activity
harvest just to make 0.9 larger. The release is represented by review-ready
packet quality, Guided Setup / Doctor, Review-loop Status, GitHub activity
harvest, redaction correctness, and release-facing docs. Further work should be
bug-driven or opened as a new non-release lane.
