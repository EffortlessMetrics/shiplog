# shiplog 0.9.0 - Release Decision

**Date:** 2026-05-17
**Decision:** keep hold
**Latest shipped release:** `v0.8.0`
**Candidate:** `v0.9.0` review-ready packet work on `main`

## Decision

Keep the `v0.9.0` release hold active.

Do not tag, publish to crates.io, create a GitHub release, or run release-install
smoke for `v0.9.0` from this decision.

The post-0.8 soak work through #397 is valuable and release-relevant, but it is
not release approval. The loop is calmer and better covered; the missing release
input is an explicit owner decision to execute the release after reviewing the
current caveats and final preflight.

## Options Considered

| Option | Decision | Reason |
| --- | --- | --- |
| A. Keep hold | selected | The loop has useful soak proof, but owner approval and final release preflight are not present. |
| B. Resume 0.9 release | rejected for now | No security fix, broken 0.8 behavior, or explicit owner instruction requires immediate release execution. |
| C. Fold more hardening into main and reassess | available | Use this if another concrete dogfood bug appears before release approval. |

## Resume Conditions

Open a future release-resume PR only when all of these are true:

- the owner explicitly approves release execution;
- `docs/release/0.9.0-release-hold.md` is updated to lift or replace the hold;
- the review-ready dogfood matrix and transcript caveats are reviewed;
- current `main` CI is green;
- package-version, package-boundary, publish dry-run, product proof, docs/share
  and runs CLI tests, clippy, policy gates, no-panic, and install smoke are rerun
  from current `main`;
- no local or remote `v0.9.0` tag exists before the approved release action.
