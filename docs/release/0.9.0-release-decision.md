# shiplog 0.9.0 - Release Decision

**Date:** 2026-05-17
**Updated:** 2026-05-18
**Decision:** keep hold; include Guided Setup / Doctor in unreleased candidate scope
**Latest shipped release:** `v0.8.0`
**Candidate:** `v0.9.0` review-ready packet plus Guided Setup / Doctor work on
`main`

## Decision

Keep the `v0.9.0` release hold active.

Do not tag, publish to crates.io, create a GitHub release, or run release-install
smoke for `v0.9.0` from this decision.

The post-0.8 soak work through #397 is valuable and release-relevant, but it was
not release approval. #398 kept the release hold active.

The Guided Setup / Doctor lane then landed through #409. That changes candidate
scope, not release posture. Because the work is now on `main` and package
version metadata remains `0.9.0`, a future `v0.9.0` release would include the
setup-readiness front door unless a later PR deliberately moves or reverts it.
That is acceptable because the lane landed quickly and cleanly, but it still
does not authorize release execution.

The missing release input is unchanged: an explicit owner decision to execute
the release after reviewing current caveats and final release preflight.

## Options Considered

| Option | Decision | Reason |
| --- | --- | --- |
| A. Keep hold and continue Guided Setup | completed, hold preserved | #399-#409 landed the bounded Guided Setup / Doctor lane while keeping release execution paused. |

| C. Expand 0.9 scope to include Guided Setup / Doctor and release later | selected for candidate scope only | Guided Setup landed quickly and cleanly, but tagging, crates.io publish, GitHub release creation, and release-install smoke remain blocked. |

## Current Release Posture

- `v0.8.0` remains the latest shipped release.
- crates.io latest remains `shiplog = "0.8.0"`.
- `v0.9.0` remains a prepared but unreleased candidate, now including
  review-ready packet quality and Guided Setup / Doctor work on `main`.
- `scripts/check-release-hold.sh` blocks held `v0.9.0` release execution.
- Manual `release.yml` dispatch requires a semver tag and
  `owner_approved_release_execution`, and the active hold still blocks
  `v0.9.0`.

## Resume Conditions

Open a future release-resume PR only when all of these are true:

- the owner explicitly approves release execution;
- `docs/release/0.9.0-release-hold.md` is updated to lift or replace the hold;
- the review-ready dogfood matrix and transcript caveats are reviewed;
- Guided Setup / Doctor caveats from #399-#409 are reviewed;
- current `main` CI is green;
- package-version, package-boundary, publish dry-run, product proof, docs/share
  and runs CLI tests, clippy, policy gates, no-panic, and install smoke are rerun
  from current `main`;
- no local or remote `v0.9.0` tag exists before the approved release action.

## What To Do Next

Do not keep adding features just to justify 0.9.

Allowed next work:

- fix concrete dogfood bugs found in the current loop;
- fix concrete setup dead ends found in doctor/source-status/guided-init flows;
- refresh wording or docs when a transcript shows user confusion;
- keep release-hold receipts current;
- prepare a future release-resume PR only after explicit approval.

Do not extend Guided Setup / Doctor just to make 0.9 larger. The lane is now
represented by #399-#409; further work should be bug-driven or deliberately
opened as a new non-release lane.
